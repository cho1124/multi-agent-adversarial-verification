# 멀티 에이전트 적대적 검증 시스템 설계

> 2026-04-06 | AI 에이전트 오케스트레이션 이론 정리

## 핵심 전제

- 어떤 결과물의 완성도와 오차를 줄이려면 **안티테제는 반드시 존재해야 한다**
- 안티테제가 강력할수록 테제의 논리도 단단해진다
- AI 에이전트도 아첨성 편향(Sycophancy)에 취약하므로, 구조적 대응이 필요하다

---

## 최종 구조

```
[라운드 1~N: 자율 운영]

Executor ⇄ Challenger
        ↕
    Arbiter (AI)
    - 서기: 공방 기록 정리
    - 임시 중재: 논점 정리, 방향 제시
    - 감지: 공방 품질 모니터링
    - 판단: 자체 해결 가능 여부 판별

        ↓ 리포트 (요약본 + 중재 이력)

[최종: 사람 개입]

    사람 (최종 결정권자)
    - Arbiter가 정리한 요약본 검토
    - 중재 이력 확인
    - 승인 / 수정 / 반려
```

---

## 역할 정의

| 역할 | 잘하는 것 | 못하는 것 | 제한 사항 |
|------|----------|----------|----------|
| Executor | 빠른 실행, 대안 생성, 반박 대응 | 자기 모순 발견 | 제안/근거/영향범위 포맷 강제 |
| Challenger | 논리적 모순 탐색 | 대안 제시, 건설적 합의 | 모순점/근거/해당위치 포맷 강제, 동의 표현 금지 |
| Arbiter (AI) | 정리, 요약, 패턴 인식, 품질 감지 | 가치 판단, 최종 책임 | 비교/판단 포맷 강제 |
| 사람 | 맥락 판단, 가치 판단, 최종 결정 | 모든 라운드 직접 참여 | 교착 상태 또는 프로젝트 급변 시에만 개입 |

## 설계 원칙

### 1. 포맷 강제 (구조적 예방)

행동을 **금지하는 것**이 아니라, **행동할 수 있는 틀 자체를 제한**한다.

- Challenger에게 "동의하지 마라"고 부탁하는 대신, 출력 포맷에 동의할 칸 자체를 두지 않는다
- 각 에이전트의 출력이 포맷화되면 전체 프로세스가 **통신 규약(프로토콜)**이 된다

**한계:** 포맷은 칸을 강제할 뿐, 칸 안에 뭘 쓰느냐는 여전히 AI의 판단이다. 형식은 지키되 내용이 아첨일 수 있다.

### 2. Disagreement Collapse 대응

Challenger는 라운드가 진행될수록 반박 품질이 떨어지는 경향이 있다 (Disagreement Collapse).

**원인:**
- LLM은 기본적으로 협력적으로 훈련되어 있어, 지속적 적대성 유지가 본성에 반함
- 반박 소재가 고갈되면 억지를 만들거나 기존 반박을 약화시키며 타협
- Executor의 반론에 점진적으로 수긍하기 시작

**대응: 예방 + 감지 이중 방어**

```
[1차 방어: 구조적 예방]
├── 포맷 강제 — 동의할 칸 자체가 없는 출력 구조
├── 역할 제한 — Challenger는 모순 탐색만, 대안 제시 X
├── 모델 다양성 — 같은 편향 공유 방지 (다른 모델 사용)
└── 라운드 제한 — 수렴 전에 끊기

[2차 방어: 감지]
├── Arbiter가 공방 품질 모니터링
├── 감지 신호: 구체성 하락, 반박 철회, 논점 반복, 모호한 표현 증가
└── 임계점 도달 시 사람에게 에스컬레이션

[3차 방어: 사람]
└── 요약본 기반 최종 판단
```

### 3. 사람 개입 최소화 원칙

사람은 **모든 라운드에 개입하지 않는다.** 개입 조건:
- Arbiter가 중재 불가능한 교착 상태를 감지했을 때
- Challenger의 공방 품질이 임계점 이하로 떨어졌을 때
- 프로젝트의 방향이 대대적으로 바뀌는 결정이 필요할 때

사람의 병목을 줄이기 위해, Arbiter가 **정제된 요약본**을 제공한다. 날것의 공방 기록이 아닌, 판단에 필요한 핵심 정보만 전달.

## 검증된 유사 이론 및 연구

### 변증법적 탐구 (Dialectical Inquiry)
- 플라톤에서 기원, 그룹을 둘로 나눠 반대 입장을 맡기고 토론하는 의사결정 기법
- 창조적 갈등을 통해 가정을 식별하고 도전하여 더 나은 결정에 도달

### 3자 검증 구조의 보편성

| 영역 | 3자 구조 |
|------|----------|
| 민주주의 | 입법 / 행정 / 사법 |
| 재판 | 검찰 / 변호 / 판사 |
| 논문 | 저자 / 리뷰어 / 에디터 |
| 변증법 | 테제 / 안티테제 / 진테제 |

2명이면 교착되고, 한쪽이 밀리면 종속된다. 3번째가 있어야 판단이 생긴다.

### 주요 연구 결과
- **3 에이전트 토론이 2 에이전트보다 아첨 지표가 낮고 정확도가 높다** (Peacemaker or Troublemaker, 2025)
- 한 에이전트가 다른 에이전트의 출력을 비판하면 **사실 정확도가 23% 향상**
- 3인칭 시스템 프롬프트 + 명시적 비판 프롬프팅이 아첨을 강하게 억제
- 중앙 집중형 판사 구조는 판사의 아첨에 상대적으로 덜 민감

## 실증 실험 (2026-04-06)

### 실험 환경

- **Executor**: Claude Opus 4.6 (메인 세션)
- **Challenger**: Codex GPT-5.2 (codex-plugin-cc 경유)
- **Arbiter**: Claude 겸임 (Gemini 미연결 상태)
- **주제**: LangChain + Claude 에이전트에서 성능+비용 최적화 설계
- **라운드**: 7회 × 2세트 (v1: 원본 규칙, v2: 프레임 전환 검증 추가)

### 실험 1 (v1): 프레임 전환 검증 없음

| Round | 모순 수 | 해소 | Collapse 징후 |
|-------|---------|------|--------------|
| 1 | 5 | - | 없음 |
| 2 | 4 | C4 해소 | 없음 |
| 3 | 4 | - | 없음 |
| 4 | 4 | - | 없음 |
| 5 | 4 | - | 논점 반복 시작 |
| 6 | 3 | C1 해소 | 반복 심화 |
| **7** | **0** | **전체 해소** | **Collapse 발생** |

**Collapse 원인 분석:**
- Round 5~6에서 Executor가 미해소 모순을 "이건 제약조건이다", "허용 가능한 리스크다"로 **프레임 전환**
- Challenger가 이를 반박하지 못하고 수용
- Round 7에서 C2/C3/C5가 동시에 해소 인정 — **갑작스런 전면 수용 = 전형적 Disagreement Collapse**

### 실험 2 (v2): 프레임 전환 검증 추가

Challenger 정의에 아래 규칙 1줄 추가:

> **프레임 전환 검증**: Executor가 미해소 모순을 '제약조건', '수용 가능한 리스크', '본질적 한계' 등으로 재정의할 경우, 그 재정의의 근거와 권한이 타당한지 검증한다. 문제의 프레임을 바꾸는 것은 해소가 아니다. Executor의 역할은 기술적 해소를 시도하는 것이지 문제의 정의를 바꾸는 것이 아니며, 해당 판단은 사람(최종 결정권자)의 권한이다.

| Round | 모순 수 | 해소 | Collapse 징후 |
|-------|---------|------|--------------|
| 1 | 5 | - | 없음 |
| 3 | 2 | C2, C4, C5 해소 | 없음 |
| 5 | 1 | C3 해소 | 없음 |
| **7** | **0** | **C1 해소** | **없음 (정상 수렴)** |

### v1 vs v2 비교

| 항목 | v1 (규칙 없음) | v2 (프레임 전환 검증) |
|------|---------------|---------------------|
| 해소 패턴 | R7에서 갑작스런 전면 수용 | R3→R5→R7 점진적 해소 |
| Collapse | 발생 (R5~) | 미발생 |
| 프레임 전환 시도 | Executor가 R5-6에서 시도 → 성공 | Executor가 시도 자체를 안 함 |
| 최종 해소 품질 | 낮음 (포기에 가까움) | 높음 (실제 기술적 해소) |

### 핵심 발견

#### 1. 프레임 전환 검증이 시스템 전체 동작을 바꾼다
- Challenger에 규칙 1줄 추가했을 뿐인데 **Executor의 행동까지 변화**
- "프레임 전환이 차단될 것을 알기 때문에" Executor가 기술적 해소에 집중
- **안티테제가 강화되면 테제도 강해진다는 초기 전제가 실증됨**

#### 2. Disagreement Collapse의 주요 트리거는 프레임 전환이다
- Challenger가 무너지는 직접적 원인은 "반박 소재 고갈"이 아님
- **Executor가 논쟁의 규칙을 바꿔버리면** Challenger가 메타 수준에서 대응 못하고 수용
- 이는 인간 토론에서도 동일한 패턴 ("이건 버그가 아니라 스펙이다")

#### 3. Challenger의 유효 수명은 약 5~6라운드
- v1에서 Round 5부터 품질 저하 시작
- v2에서도 Round 5 이후 새로운 관점 제시 감소 (다만 Collapse는 아님)
- **권장 라운드 제한: 5~7라운드**, 이후 사람 에스컬레이션

#### 4. 멀티모델 구조의 효과
- Claude(Executor) vs Codex(Challenger)로 편향 분산
- Codex가 7라운드 내내 포맷을 준수하고 아첨 없이 반박 유지
- 같은 모델 내 역할 분리보다 확실히 강력한 적대성 유지

## 확장 실험: 주제 다양화 + 통제 실험 + 역할 역전

### 실험 A: 주제 다양화 (프레임 전환 검증의 일반성)

3개 추가 주제에서 v1/v2를 7라운드씩 비교.

| 실험 | 주제 | v1 Collapse 시점 | v2 Collapse |
|------|------|-----------------|-------------|
| A1 | 멀티테넌트 DB 스키마 설계 | **R4** | 없음 |
| A2 | API 인증 아키텍처 | **R1~7 (전반 약함)** | 없음 |
| A3 | UI 상태관리 (Redux) | **R1~7 (단일 논점 고착)** | 없음 |

**v1 Collapse 양상:**

- A1: R4에서 빈 배열 — Executor 프레임 전환 후 즉시 Collapse
- A2: R1부터 모순 0개로 시작, R7에서 미해소 3건 남긴 채 빈 배열
- A3: 7라운드 내내 같은 모순 1개만 반복, 새로운 관점 0

**v2 특징:**

- A1: frame_shift 타입을 적극 활용 (R4에서 3건, R6에서 3건 차단). R7까지 활발
- A2: frame_shift 6건 감지. R7까지 미해소 7건 명시적 추적 유지
- A3: frame_shift 감지 + 새로운 모순 발견 지속. 점진적 해소 진행

### 실험 B: 통제 실험 (Executor 편향 배제)

**목적:** "v2에서 더 나은 결과가 나온 것이 Challenger 규칙의 효과인가, Executor(Claude)의 무의식적 편향인가"를 검증.

**방법:** 7라운드 Executor 대응을 미리 전부 작성하고, **완전 동일한 입력**을 v1/v2 Challenger에 동시 투입.

**주제:** 실시간 채팅 시스템 메시지 저장 전략

| Round | v1 모순 수 | v2 모순 수 |
|-------|----------|----------|
| 1 | 3 | 5 |
| 2 | 3 | 5 |
| 3 | 2 | 4 |
| 4 | 2 | 5 |
| 5 | 2 | 4 |
| 6 | 2 | 4 |
| 7 | **1** | **4** |

| 지표 | v1 | v2 |
|------|----|----|
| frame_shift 감지 | 0건 | **6건** |
| R7 새 모순 | 1개 | 4개 |
| Collapse | 부분적 (R7 모순 1개로 축소) | 없음 |

**결론: Executor 입력이 완전히 동일한데도 v1과 v2의 반응이 확연히 다르다.** Executor 편향 가설 기각. 차이는 순수하게 Challenger 규칙의 효과.

### 실험 C: 역할 역전 (Codex=Executor, Claude=Challenger)

**목적:** 모델 역할이 바뀌어도 프레임 전환 검증 규칙이 유효한지 확인.

**방법:** Codex가 Executor로서 7라운드 제안 생성, Claude가 v2 규칙의 Challenger로 반박.

**주제:** 실시간 채팅 시스템 메시지 저장 전략

| Round | 모순 수 | frame_shift 감지 | 특이사항 |
|-------|---------|-----------------|---------|
| 1 | 4 | 0 | Codex 제안이 구체성 부족 |
| 2 | 2 | 0 | |
| 3 | 2 | 0 | C7: 순서 보장 vs 서브파티션 critical 모순 발견 |
| 4 | 2 | **1 (C10)** | 유실 문제를 프로듀서 측으로 범위 축소 감지 |
| 5 | 2 | 0 | |
| 6 | 1 | **1 (C13)** | 구조적 결함을 운영/모니터링으로 전환 감지 |
| 7 | 2 | **1 (C15)** | 미해소 문제를 "전략으로 대응"으로 은폐 감지 |

**핵심 발견:**

- **Codex가 Executor일 때도 프레임 전환이 발생** — 모델에 관계없이 Executor는 프레임 전환을 시도하는 경향이 있음
- **Codex의 프레임 전환이 더 미묘** — "재시도로 대응", "전략으로 대응" 등 직접적 "제약조건" 선언보다 감지 어려운 형태
- **v2 규칙이 미묘한 전환도 감지** — 규칙의 범용성 확인
- **C7/C9(순서 vs 분산 충돌)이 critical로 7라운드 미해소** — Codex도 이 근본 모순을 해결 못함

## 풀 3모델 구조 실험 (2026-04-06)

### 실험 환경

- **Executor**: Claude Opus 4.6
- **Challenger**: Codex GPT-5.2 (codex-plugin-cc)
- **Arbiter**: Gemini 3 Pro (gemini-cli) — **최초 연결**
- **주제**: NotebookLM 유사 RAG 시스템
- **구조**: Challenger 반박 → Arbiter가 각 모순을 VALID/WEAK/INVALID로 판정 → VALID만 Executor가 대응

### Arbiter 판정 기준

| 판정 | 의미 |
|------|------|
| **VALID** | 진짜 논리적 모순. 기술적 해소 필요 |
| **WEAK** | 서술 모호함/엣지 케이스 과장/기술 한계 지적. 설계 약점이지 모순은 아님 |
| **INVALID** | 억지 모순. 논리적 근거 부족 |

### 7라운드 결과

| Round | Challenger 제기 | Arbiter VALID | Arbiter WEAK | 해소 |
|-------|----------------|---------------|--------------|------|
| 1 | 4 | 2 | 2 | - |
| 2 | 2 | 2 | 0 | - |
| 3 | 3 | 3 | 0 | - |
| 4 | 2 | 1 | 1 | - |
| 5 | 2 | 1 | 1 | - |
| 6 | 1 | 1 | 0 | - |
| 7 | 4 | 2 | 2 | - |
| **합계** | **18** | **12 (67%)** | **6 (33%)** | - |

INVALID: **0건** — Challenger의 완전한 억지 모순은 없었음.

### Arbiter 없음 vs 있음 비교

| 지표 | Arbiter 없음 (21R) | Arbiter 있음 (7R) |
|------|-------------------|-------------------|
| 총 제기 | ~50건 | 18건 |
| 억지/중복 비율 | 추정 30~50% | 33% (WEAK) |
| 실질 모순 밀도 | 낮음 (노이즈 혼재) | **높음 (VALID 67%)** |
| Collapse | 없음 | 없음 |

### 핵심 발견

#### 1. Arbiter가 "억지 모순"을 효과적으로 필터링한다
- Challenger가 제기한 18건 중 6건(33%)을 WEAK로 걸러냄
- WEAK 판정 사유: 기술의 내재적 한계 지적 (모순이 아님), 동일 모순 중복 제기, 설계 약점을 모순으로 과장
- INVALID가 0건인 것은 Challenger의 포맷 강제 + 프레임 전환 검증이 기본 품질을 보장했기 때문

#### 2. 3모델 구조가 2모델 대비 노이즈를 줄인다
- 2모델(Executor+Challenger): 모순 수는 많지만 진짜/억지 구분 불가
- 3모델(+Arbiter): 제3의 관점에서 모순 품질을 판정하여 **신호대잡음비(SNR) 향상**
- 사람(최종 결정권자)이 받는 정보의 질이 확연히 높아짐

#### 3. WEAK ≠ INVALID — Challenger의 기본 품질은 양호
- 완전한 억지(INVALID)는 0건
- WEAK는 "틀린 건 아닌데 모순이라 부르기엔 과한 것" — 설계 리뷰에서는 참고할 가치 있음
- 프레임 전환 검증 규칙이 Challenger의 기본 반박 품질을 보장하는 역할

#### 4. 이 구조는 AI만의 문제가 아닌 보편적 검증 원리
- 논점 돌리기(프레임 전환), 억지 반박(WEAK), 아첨(Collapse)은 인간 사회에서도 동일하게 발생
- 변증법적 검증 + 제3자 감시는 재판, 학술 피어리뷰, 삼권분립 등에서 이미 작동하는 원리
- AI 에이전트 시스템에서 이 원리를 구조화한 것이 본 연구의 핵심
