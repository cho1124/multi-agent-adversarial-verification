# 전광판 영상 동기화 시스템 적대적 검증 실험 (2026-04-07)

> 여러 PC에 설치된 전광판 영상 플레이어가 같은 영상을 동시에 재생할 때, 화면이 어긋나지 않도록 맞추는 "동기화 시스템"의 문제점을 찾고 개선안을 설계한 실험

---

## 결론

### 실무 결론: "핵심 결함은 네트워크가 아니라 시간 기준 설계"

현재 시스템의 가장 큰 문제는 TCP 지연이나 290ms 딜레이 같은 표면적인 것이 아니라, **"서로 다른 PC의 시계를 어떻게 맞추는가"**라는 근본 설계가 빠져 있었다는 것.

| 지표 | 현재 시스템 | 개선 후 설계 |
|------|-----------|-------------|
| 시간 기준 | 없음 (서버 재생 시각만 전송) | PC별 정밀 시계(QPC) + 인터넷 시계(NTP) 조합 (**단, PC 간 매핑 규칙 미완성**) |
| 어긋남 보정 | 영상 다시 로딩 (화면 끊김) | 3단계 분기: 약간 어긋남 → 속도 미세 조정 / 많이 어긋남 → 다시 로딩 |
| 딜레이 보상 | 고정값 290ms (사람이 직접 조정) | 5단계로 쪼개서 자동 측정 |
| 허용 오차 | 고정값 80ms (사람이 직접 조정) | 자동 상태 전환 (들어갈 때와 나올 때 기준을 다르게 = 떨림 방지) |
| 통신 방식 | TCP (안정적이지만 느림) | UDP 멀티캐스트 (빠르지만 유실 가능 → 유실 대비 로직 추가) |
| 50ms 이하 떨림 | 원인 모름 | 통계 분석으로 "잡음인지 실제 밀림인지" 구분 |
| 모니터링 | 서버 화면에서 눈으로 확인 | 자동 로그 + 오차 분포 그래프 + 외부 카메라 측정 |

### 미해소 문제 (사람이 판단해야 할 것들)

| 문제 | 설명 | 왜 중요한가 |
|------|------|-----------|
| **PC 간 시계 매핑** | 각 PC의 정밀 시계(QPC)는 그 PC 안에서만 유효. 서버 시계와 클라이언트 시계를 같은 축에 놓는 변환 규칙이 없음 | **이게 해결 안 되면 나머지 모든 보정이 무의미** |
| **속도 조정 미지원** | 일부 영상 재생기는 재생 속도 변경을 지원 안 함. 이 경우 "약간 어긋남" 상태에서 할 수 있는 게 없음 | 상태머신의 핵심 동작이 특정 환경에서 작동 안 함 |
| **반복 재생 아닌 영상** | 콘텐츠 전환 규칙이 "다음 반복 시작점에서 전환"에만 의존. 한 번만 재생하는 영상은 전환 시점이 없음 | 비반복 콘텐츠 미지원 |
| **복구 시 끊김** | 재시작 후 즉시 재생하면서 동시에 재로딩으로 동기 맞추기 = 논리적 모순 (재로딩 중에는 재생이 끊김) | "끊김 없는 복구"가 구조적으로 불가능 |

### 이 실험에서 확인된 것

1. **독립 체크리스트가 핵심 누락을 사전에 잡아냄** — 25개 체크 항목 중 "시계 기준 정의"가 7라운드 내내 해소 안 됨 → 가장 근본적인 문제였음을 증명
2. **논점 돌리기 감지 작동** — "NTP 대신 PTP 쓰면 되지 않나?"(원인은 안 찾고 도구만 바꾸기), "재시작 후 기다리면 되지"(서비스 중단을 허용 가능으로 재정의) 2건 감지
3. **Executor(제안자) 붕괴 감지** — R4부터 Arbiter가 "제안자의 설계가 근본적으로 무너졌다"고 판정. 제안자도 검증 대상이 되는 대칭 구조의 효과
4. **Arbiter 판정 정확도** — 전체 42건 중 34건(81%)이 진짜 모순, 8건(19%)이 약한 지적, 억지 0건

### 한계

1. PC 간 시계 매핑의 해결책을 끝내 못 찾음 — 추가 연구 필요
2. Windows 환경에서만 논의됨 — Linux 등 다른 OS 미검증
3. 실제 전광판 하드웨어에서 테스트하지 않음 — 이론 설계 단계

## 최종 확정 설계 (일부 미완성)

### 확정된 부분

**3단계 상태 전환 (어긋남 정도에 따라 자동 분기):**

```
[정상] 오차 50ms 미만
   │ 오차가 커지면 (100ms 초과)
   ↓
[약간 어긋남] 오차 50~200ms → 재생 속도를 살짝 빠르게/느리게 조정
   │ 오차가 더 커지면 (200ms 초과)
   ↓
[크게 어긋남] 오차 200ms 초과 → 영상을 다시 로딩해서 맞춤

   * 떨림 방지: "약간 어긋남"으로 들어가는 기준(100ms)과
     "정상"으로 돌아오는 기준(30ms)을 다르게 설정
```

**딜레이 5단계 분해 (290ms 한 덩어리 대신):**

```
동기 신호 수신 → [네트워크 지연] → 오차 계산
영상 다시 로딩 → [디코더 준비] → [버퍼 채우기] → [첫 프레임 표시] → [화면 반영] → 재생
```

각 단계를 개별로 측정해서 어디가 병목인지 파악.

**50ms 이하 떨림 진단:**

최근 60초간의 오차 패턴을 통계 분석:
- **무작위 패턴(잡음)** → 무시해도 됨. 떨림 방지 설정으로 충분
- **한쪽으로 밀리는 패턴(드리프트)** → 재생 속도를 미세 조정해서 따라감

**반복 재생 동기:** 영상이 처음으로 돌아갈 때 "LOOP_START" 신호를 3번 반복 전송. 누적된 어긋남을 강제 초기화.

**콘텐츠 확인:** 서버가 동기 신호에 영상 파일의 ID와 해시값을 포함. 다른 영상이 재생 중이면 백그라운드에서 다운로드 후 전환.

### 미완성 (해결 못한 것들)

- **PC 간 시계 매핑**: 각 PC의 정밀 시계(QPC)를 서로 비교할 수 있게 만드는 변환 규칙
- **속도 조정 미지원 시 대안**: 재생 속도 변경이 안 되는 영상 재생기에서의 동작
- **비반복 콘텐츠**: 한 번만 재생하는 영상의 전환 시점
- **복구 시 끊김 없애기**: 재시작 후 즉시 재생 + 동기 맞추기를 동시에 하는 방법
- **외부 측정**: 실제 화면을 카메라로 찍어서 어긋남을 측정하는 방법 (QPC 타임코드는 PC 간 비교 불가)

## Challenger 독립 체크리스트

Challenger가 제안을 보기 전에 "이 주제에서 반드시 다뤄야 할 것" 25항목을 독립 생성.

핵심 항목:
- **CL1** — 서버와 각 PC의 시계를 어떻게 맞추는가? (NTP/PTP 사용 여부)
- **CL2** — 재생 시각의 기준이 뭔가? (OS 시계? 내부 타이머? 디코더 타임라인?)
- **CL3** — 네트워크 지연/떨림/패킷 손실을 측정하고 있는가?
- **CL7** — 영상 재로딩 말고 다른 보정 방법(속도 조정 등)이 있는가?
- **CL8** — 50ms 이하에서 계속 어긋나는 원인이 잡음인지 실제 밀림인지 구분 가능한가?
- **CL11** — 재로딩 과정의 각 단계별 소요 시간을 측정하는가?
- **CL13** — 모니터 출력 지연(화면 주사율, GPU 처리)이 최종 표시에 미치는 영향은?
- **CL24** — 실제 화면을 외부 장비(고속 카메라 등)로 촬영하여 어긋남을 검증하는 수단이 있는가?

## 핵심 전환점

### R4 — "서로 다른 PC의 시계를 직접 비교할 수 없다"

Executor가 "모든 시각을 QPC(Windows의 정밀 내부 시계)로 통일하겠다"고 제안. Challenger가 "QPC는 각 PC 안에서만 유효한 시계라서, 서버의 QPC 값과 클라이언트의 QPC 값을 직접 빼서 비교할 수 없다"고 지적.

이 모순이 R7까지 해소되지 않아 Arbiter가 **Executor(제안자)의 설계가 무너졌다**고 판정.

**왜 이게 중요한가:** 동기화의 핵심은 "두 PC의 시계를 같은 기준에 놓는 것". 이게 안 되면 아무리 좋은 보정 알고리즘을 만들어도 입력 자체가 틀린 셈.

### R2 — 논점 돌리기 2건 감지

1. **"NTP 대신 PTP 쓰면 되지 않나?"** — 현재 어긋남의 원인이 시계 정밀도 때문인지 확인도 안 했는데, 더 정밀한 도구로 바꾸자고 제안. Arbiter가 "원인을 찾지 않고 도구만 교체하는 건 논점 돌리기"로 판정.

2. **"재시작 후 동기 신호 올 때까지 기다리면 되지"** — 전광판이 재시작됐을 때 화면이 안 나오는 시간을 "기다리면 된다"로 처리. Arbiter가 "서비스 중단 문제를 허용 가능하다고 재정의한 것이지 해결한 것이 아니다"로 판정.

## 현재 시스템 vs 개선 설계 비교

| 영역 | 현재 시스템 | 개선 설계 |
|------|-----------|----------|
| 어긋남 원인 파악 | 불가능 (290ms 한 덩어리) | 5단계 분해로 병목 식별 |
| 보정 방식 | 무조건 재로딩 → 끊김 | 약간: 속도 조정, 심함: 재로딩 |
| 50ms 이하 떨림 | 허용치 올려서 포기 | 잡음 vs 밀림 통계 분석 |
| 통신 | TCP (재전송으로 신호 지연) | UDP 멀티캐스트 (빠른 전달) |
| 파라미터 조정 | 사람이 시행착오 | 시스템이 자동 적응 |
| 모니터링 | 서버 화면 육안 확인 | 자동 로그 + 분포 분석 |
| 반복 재생 경계 | 처리 없음 | LOOP_START 3회 전송 + 강제 재동기 |
