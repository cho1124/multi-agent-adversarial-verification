# 고정 딜레이 → 단계별 계측

> R1-R4에 걸쳐 290ms 고정값의 근본 한계 식별 + 5단계 분해

## 문제

현재 시스템은 동기화 시 **고정 290ms 딜레이**를 추가하여 로딩 시간을 보상. 이 값은 콘텐츠/하드웨어와 무관하게 동일 적용되어, 실제 로딩 시간과 불일치 시 오차가 발생.

## 적대적 검증에서 일어난 일

**R1-C6** — Challenger: "EMA로 로딩시간을 추정해도 오차의 원인을 분리할 수 없다. 네트워크, 디코더, 렌더링 중 어디가 병목인지 모른다"

**R1-C9** — Challenger: "재로딩 단계별 소요 시간 계측이 없으면 290ms의 오판 원인을 해부할 수 없다"

→ Executor가 EMA 단일 지표를 폐기하고 5단계 분해로 전환

## 최종 설계

```
동기 신호 수신 → [network_delay_ms] → 오차 계산
재로딩 시작 → [decode_init_ms] → [buffer_fill_ms] → [first_frame_ms] → [display_delay_ms] → 재생
```

각 단계에 QPC 타임스탬프를 찍어 정밀 계측.

## 개선된 점

| 항목 | 개선 전 | 개선 후 |
|------|--------|--------|
| 딜레이 보상 | 고정 290ms | **단계별 실측 기반** |
| 병목 식별 | 불가 | 5단계 중 지배적 단계 식별 가능 |
| 콘텐츠별 차이 | 무시 | 단말별/콘텐츠별 프로파일링 |
| 조정 방식 | 수동 (스케줄 파일) | 자동 계측 + 적응 |
