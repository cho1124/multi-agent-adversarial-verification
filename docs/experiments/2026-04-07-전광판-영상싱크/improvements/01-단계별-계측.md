# 고정 딜레이 → 단계별 측정

> R1-R4에 걸쳐 "290ms 한 덩어리"의 한계를 발견하고 5단계로 분해

## 문제

현재 시스템은 영상을 다시 로딩할 때 **고정값 290ms를 더해서** 타이밍을 맞춤. 하지만 실제 로딩 시간은 영상 크기, PC 성능, 네트워크 상태에 따라 다르기 때문에, 이 고정값이 맞을 때도 있고 안 맞을 때도 있음.

## 단독 설계였다면

"290ms가 적당하지 않으면 스케줄 파일에서 수동으로 조정하세요" — 매뉴얼에 실제로 이렇게 적혀 있음. 사람이 시행착오로 값을 찾아야 하는 구조.

## 적대적 검증에서 일어난 일

**R1** — Challenger: "처음에 EMA(지수이동평균)로 로딩 시간을 추정하겠다고 했는데, 로딩 시간 안에는 네트워크 지연, 디코더 준비, 화면 표시 지연이 전부 섞여 있다. 하나의 숫자로 추정하면 어디가 병목인지 모른다"

**R4** — 단일 지표(EMA)를 폐기하고 **5단계 분해**로 전환

## 최종 설계

```
동기 신호 수신 → [네트워크 지연: 몇 ms?]
   → 오차 계산
영상 다시 로딩 → [디코더 준비: 몇 ms?]
   → [버퍼 채우기: 몇 ms?]
   → [첫 프레임 표시: 몇 ms?]
   → [화면에 실제 반영: 몇 ms?]
   → 재생 시작
```

각 단계마다 시간을 찍어서 **어디서 가장 오래 걸리는지** 파악.

## 뭐가 좋아졌나

| 항목 | 이전 | 이후 |
|------|------|------|
| 딜레이 보상 | 290ms 고정 (사람이 수동 조정) | **5단계 자동 측정** |
| 병목 파악 | 불가능 | 어떤 단계가 느린지 바로 확인 |
| 콘텐츠별 차이 | 무시 | PC별/영상별로 개별 측정 |
| 조정 방식 | 사람이 시행착오 | 시스템이 자동 적응 |
