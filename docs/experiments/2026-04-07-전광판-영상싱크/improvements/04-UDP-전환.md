# TCP → UDP 멀티캐스트

> R1-R5에 걸쳐 동기 신호 전송 방식을 변경

## 문제

현재 시스템은 동기 신호를 **TCP**로 보냄. TCP는 "데이터가 반드시 도착하는 것"을 보장하는 대신, 중간에 패킷이 유실되면 **자동으로 다시 보내고 기다림**. 이 재전송 대기 시간이 동기 신호를 불규칙하게 지연시킴.

쉽게 말해: TCP는 "늦더라도 확실히" 도착시키는 방식인데, 동기 신호는 **"확실하진 않아도 빨리"** 도착해야 의미가 있음. 1초 전 동기 신호가 늦게 도착해봤자 이미 쓸모없음.

## 단독 설계였다면

"TCP가 안정적이니까 TCP 쓰면 되지" — 실시간 신호의 특수성을 고려하지 않음.

## 적대적 검증에서 일어난 일

**R1-C3** — Challenger: "UDP로 바꾸자고 했는데 유실/중복/순서 뒤바뀜 처리 규칙이 없다"

**R2-C2** — Challenger: "신호 유실 시 마지막 상태를 유지하겠다고 했는데, 유지하는 동안 시계가 계속 밀리면?"

**R5** — "마지막 상태 유지" 방식을 폐기. 3연속 유실 시 최근 밀림 추세를 기반으로 **자율 보정 모드** 진입.

## 최종 설계

- **UDP 멀티캐스트** — 서버가 한 번 보내면 모든 클라이언트가 동시에 수신 (TCP처럼 개별 연결 불필요)
- **메시지 내용**: 순번, 서버 시각, 영상 ID, 영상 해시값, 현재 재생 위치
- **유실 대응**: 3연속 유실 → 최근 밀림 추세 기반 자율 보정 / 신호 복구 시 즉시 서버 기준으로 재동기

## 뭐가 좋아졌나

| 항목 | 이전 (TCP) | 이후 (UDP) |
|------|-----------|-----------|
| 신호 지연 | 재전송 대기로 불규칙 | 최소 지연 (재전송 없음) |
| 유실 시 | TCP가 알아서 재전송 (늦게라도 도착) | 유실 인정 + 자율 보정 |
| 클라이언트 수 증가 | 개별 연결 필요 (부하 증가) | 멀티캐스트 1회 전송 (부하 고정) |
| 신호 순서 | TCP가 보장 | 순번으로 직접 관리 |
