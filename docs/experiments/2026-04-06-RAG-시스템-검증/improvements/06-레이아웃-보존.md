# 레이아웃 정보 보존

> 주제1 (청크/임베딩) — R5에서 발견된 파싱 단계 정보 소실 문제

## 문제

PDF를 파싱하면 텍스트만 추출되고 **표 구조, 단 배치, 이미지 위치 등 레이아웃 정보가 사라짐**. 이 정보 없이는 "이 텍스트가 표 안의 데이터인지, 본문인지" 구분이 안 되어 청킹과 인용 정확도가 떨어짐.

## 단독 설계에서는

텍스트 추출만 수행. 레이아웃 정보를 활용하겠다는 발상 자체가 없음.

## 적대적 검증에서 일어난 일

**R3** — Executor가 문서를 `[structured|unstructured]×[table_heavy|text_heavy]`로 분류하겠다고 제안.

**R3** — Challenger: "상호배타 분류를 폐기했다면서 축 내부는 여전히 이항이다"

이 지적으로 파이프라인이 **속성 태그 기반**으로 전환됨. 그런데:

**R5** — Challenger: "파싱 단계에서 텍스트로 변환하면 레이아웃 정보가 소실되어, 구간 분석에 필요한 structure_ratio나 table_ratio를 계산할 수 없다"

→ 텍스트 추출과 레이아웃 추출을 **동시에** 수행하는 구조로 재설계.

## 최종 설계

```
파싱 출력:
{
  segments: [
    {
      page: 3,
      section: "2.1 시스템 구성",
      text: "본문 텍스트...",
      layout_metadata: {
        structure_ratio: 0.7,
        table_ratio: 0.3,
        has_images: true
      },
      ocr_confidence: 0.95
    }
  ]
}
```

## 개선된 점

| 항목 | 개선 전 | 개선 후 |
|------|--------|--------|
| 파싱 출력 | 텍스트만 | 텍스트 + 레이아웃 메타데이터 |
| 문서 구간 분석 | 불가 | structure_ratio, table_ratio 자동 계산 |
| 청킹 전략 | 모든 문서 동일 | **구간 속성 기반** 적응 분할 |
| 인용 위치 | 텍스트 offset만 | 페이지 + 섹션 + 레이아웃 컨텍스트 |

## 파이프라인 재설계의 시발점

이 발견이 R3의 분류 체계 변경 → R4의 전처리 단계 추가 → R5의 레이아웃 보존 → R6의 구간별 스키마 → R7의 분할 방식 변경까지 **연쇄 재설계**를 유발. 단독 설계에서는 초기안이 그대로 유지되었을 것.
