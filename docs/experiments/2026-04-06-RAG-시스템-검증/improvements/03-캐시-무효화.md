# 캐시 무효화

> 주제3 (캐싱/인덱싱) — 3계층 동시 무효화 확정

## 문제

문서가 수정되었는데 **캐시에 옛날 답변이 남아** 사용자에게 틀린 정보를 반환하는 문제. RAG에서 캐시는 성능을 위해 필수지만, 무효화를 잘못하면 **정확도를 희생**.

## 단독 설계에서는

캐시 자체가 없음. 매번 전체 파이프라인을 재실행 → 비용 폭증.

## 적대적 검증에서 일어난 일

**R1** — Challenger: "L1 캐시에 TTL을 걸었는데, L2/L3가 먼저 무효화되면 L1이 구 결과를 반환한다. 계층 간 무효화 순서가 정의되지 않았다"

**R1** — Challenger: "semantic 캐시(코사인 유사도)의 키에 검색 설정 해시가 없다. 검색 파라미터가 바뀌면 다른 결과인데 같은 캐시를 반환한다"

**후속 라운드** — 권한 변경, 문서 삭제, 테넌트 격리 등 캐시가 깨지는 시나리오가 연쇄적으로 발견됨.

## 최종 설계

| 계층 | 키 구성 | 무효화 조건 |
|------|--------|------------|
| L1 exact | 정규화+SHA256 | 문서 갱신 시 **즉시** |
| L1 semantic | 코사인 > 0.95 + **검색설정 해시** | 문서 갱신 시 **즉시** |
| L2 검색 | reranker 이전/이후 분리 | 문서 갱신 시 **즉시** |
| L3 생성 | 청크 리스트 해시 + 모델 + 프롬프트 버전 | 문서 갱신 시 **즉시** |

핵심: **3계층 동시 무효화**. 부분 무효화는 허용하지 않음.

## 개선된 점

| 항목 | 개선 전 | 개선 후 |
|------|--------|--------|
| 캐시 존재 | 없음 (매번 재계산) | 3계층 캐시 |
| 무효화 | 해당 없음 | 문서 갱신 시 전계층 동시 무효화 |
| 일관성 | 항상 최신 (비용 대가) | 캐시 히트 시에도 최신 보장 |
| semantic 캐시 오류 | 해당 없음 | 검색설정 해시로 파라미터 변경 감지 |
