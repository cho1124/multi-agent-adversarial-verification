# 벡터 차원 SQL/Config 불일치

> 소주제1 (아키텍처 검증) — R1에서 발견, CRITICAL

## 문제

SQL 스키마와 Python 설정이 서로 다른 임베딩 모델·차원을 가정하여 **런타임에서 반드시 크래시**하는 결함.

## 단독 구현에서는

개발 중 SQL을 먼저 작성(gte-small, 384d)하고 나중에 Config를 바꿨지만(text-embedding-3-small, 1536d), SQL 업데이트를 누락. 실행해보기 전까지 발견 불가.

## 적대적 검증에서 일어난 일

**R1 — Challenger 즉시 지적:**

```
setup_v3.sql: VECTOR(384)         ← gte-small 기준
config.py:    embedding_dimensions = 1536  ← text-embedding-3-small
config.py:    embedding_model = "text-embedding-3-small"
```

Executor가 "SQL 한 줄 바꾸면 된다"고 대응했으나:

**R2 — Challenger 재반박:**

- HNSW 인덱스가 384차원으로 생성됨 → 1536으로 변경 시 인덱스 재생성 필요
- 기존 임베딩 데이터 전체 재계산 필요
- "SQL 한 줄"이 아니라 **데이터 마이그레이션** 문제

Arbiter: VALID — 인덱스+데이터 재생성은 사실

## 1차 설계와의 괴리

| 항목 | 1차 설계 확정 | 실제 구현 |
|------|-------------|----------|
| 임베딩 모델 | bge-m3 (1024d) | text-embedding-3-small (1536d) |
| SQL 스키마 | — | VECTOR(384) (gte-small) |
| 차원 | 1024 | 384 vs 1536 (이중 불일치) |

설계(1024) → 구현에서 **두 번** 변질: SQL은 384로, Config는 1536으로 각각 다르게 변경.

## 개선된 점

수정 시 반드시 포함해야 할 사항:
1. SQL과 Config의 차원을 **하나의 source of truth**에서 관리
2. 앱 시작 시 `SELECT vector_dimension FROM pg_type` 등으로 런타임 검증
3. 차원 변경 시 마이그레이션 스크립트 제공 (인덱스 재생성 + 임베딩 재계산)
