# 토큰 추정 결함

> 소주제1 (아키텍처 검증) — R1에서 발견, HIGH

## 문제

토큰 수를 `len(text) // 3`으로 추정하여, 한국어+영어 코드 혼합 콘텐츠에서 **±50% 오차**가 발생. context truncation → 답변 품질 저하로 직결.

## 단독 구현에서는

"대충 3으로 나누면 비슷하겠지"로 구현. 면접위키가 한국어 설명 + 영어 코드 블록 혼합이라 오차가 더 심해지는 구조.

## 적대적 검증에서 일어난 일

**R1 — Challenger 정량 분석:**

```python
# 현재 구현 (agents/search.py:99)
def _estimate_tokens(text):
    return len(text) // 3

# 실제 토큰 수 (tiktoken 기준)
한국어 "면접에서 자주 나오는 질문"  → 12자 / 3 = 4 (추정) vs 8 (실제) → 50% 과소
영어 "async/await pattern"        → 19자 / 3 = 6 (추정) vs 4 (실제) → 50% 과대
```

면접위키의 전형적 청크(한국어 설명 + 코드 블록):
- 한국어 부분: 과소 추정 → 토큰 예산 남음 → 불필요한 청크 포함
- 코드 부분: 과대 추정 → 조기 truncation → 코드 잘림

Arbiter: VALID — ÷3이 tiktoken 대비 ±50% 오차는 정량적 사실

## 개선된 점

```python
# 교체안
import tiktoken

enc = tiktoken.encoding_for_model("gpt-4o-mini")

def estimate_tokens(text: str) -> int:
    return len(enc.encode(text))
```

- tiktoken은 순수 Python + Rust backend, pip 설치 가능
- 로컬 모델(Ollama) 전환 시에도 tokenizer 교체만으로 대응
- 비용: 함수 호출 오버헤드 ~0.1ms (무시 가능)
