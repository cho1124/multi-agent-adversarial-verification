# 캐시 키 정합성 문제

> 소주제1 R3 C18,C19 + R4 C22,C23 + R1 C14, VALID, **3모델만 발견**

## 문제

3개 캐시 계층 모두 키가 불충분하여 잘못된 응답을 반환하거나 수정 사항이 적용되지 않는 문제.

| 캐시 | 현재 키 | 누락 | 결과 |
|------|---------|------|------|
| L1 | SHA256(query) | complexity, model, prompt version | 다른 모델의 응답 반환 (C23) |
| L2 | query + category | top_k, threshold, weight | 설정 변경해도 30분간 구결과 (C18) |
| L3 | sorted(chunk_ids) + model | 점수 순서 | 인용번호 오매핑 (C14) |

## 단일모델에서는

"캐시 전역 flush" 문제만 지적. 키 자체의 불충분성은 미발견.

## 3모델에서 발견된 과정

- R1 C14: Codex가 `generation_cache.py:20-27`의 sorted()와 `search.py:62-96`의 점수순 인용 부여를 교차 분석
- R3 C18: Codex가 `workflow.py:127-138`의 L2 키 입력이 `{"category": None}`뿐임을 확인
- R3 C19: L1 키가 query만이라 complexity/model 변경이 반영 안 됨
- R4 C22: 캐시 버전업이 로직 변경보다 선행해야 한다는 의존성 순서 발견
- R4 C23: L1 키에 complexity/model_name 필수 포함

## 수정 (의존성 순서: P1-Phase A)

```python
# L1: query + complexity + model + prompt_version
key = sha256(f"{normalize(query)}:{complexity}:{model_name}:{PROMPT_VERSION}")

# L2: query + category + retrieval_settings
key = sha256(f"{normalize(query)}:{category}:{top_k}:{threshold}:{vec_weight}")

# L3: ordered chunk_ids (sorted 아닌 점수순) + model + prompt_version
key = sha256(f"{':'.join(chunk_ids_by_score)}:{model}:{PROMPT_VERSION}")
```

캐시 키 버전업은 citation 강제, gate 분리 등 로직 변경보다 **반드시 선행** (C22).
