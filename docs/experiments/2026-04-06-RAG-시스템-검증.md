# RAG 시스템 적대적 검증 실험 (2026-04-06)

> 개선된 오케스트레이션 구조(독립 체크리스트 + 대칭적 검증 + Collapse 조작적 정의 + 프레임 전환 검증)의 실증 실험

---

## 결론

### 실무 결론: "적대적 검증은 설계 품질을 근본적으로 바꾼다"

| 지표 | 단독 설계 (검증 없음) | 적대적 검증 (3모델) |
|------|---------------------|-------------------|
| 파이프라인 재설계 횟수 | 0회 (초기안 그대로) | **3회** (구조적 결함 발견 시마다) |
| 청크 ID 설계 변경 | 0회 | **7회** (8라운드에 걸친 점진적 개선) |
| 인용 정합성 | 없음 | span 3단계 폴백 + 2단계 IoU |
| 캐시 무효화 | 없음 | 3계층 전계층 동시 무효화 |
| 환각 방지 | 프롬프트 수준 | sufficiency gate + 후검증 |
| 비용 통제 | API 종속 (통제 불가) | 로컬 우선 ~$15/월 |

**핵심:** 단독 설계에서 "이렇게 하면 되겠지"하고 넘어간 부분들이 Challenger에 의해 전부 드러났다. 특히 청크 ID 안정성은 8라운드에 걸쳐 7번 재설계될 정도로 깊은 구조적 결함이었지만, 단독 설계에서는 **발견 자체가 불가능**했다.

### 검증된 것

1. **프레임 전환 검증 규칙이 실전에서 작동한다** — 주제2 R3에서 Executor가 IoU 평가 기준을 parent로 바꿔 정밀도 저하를 은폐하려 했으나, Arbiter가 감지 → Executor 인정 → 기술적 해소로 이어짐
2. **독립 체크리스트가 생존자 편향을 완화한다** — 3개 주제에서 총 84항목 생성. Executor가 다루지 않은 영역을 사전에 가시화
3. **Collapse 조작적 정의가 종료 판정을 객관화한다** — 3개 주제 모두 rational_consensus로 정상 종료 판정. "진짜 합의인가 포기인가"를 구분 가능
4. **VALID 비율 98%** — 이전 실험(67%)에서 독립 체크리스트 도입 후 억지 모순이 거의 사라짐
5. **멀티모델 구조가 편향을 분산한다** — Claude/Codex/Gemini 각각이 다른 관점에서 동일 설계를 검토

### 검증되지 않은 것 (한계)

1. **표본 부족** — 3개 소주제, 각 1회 실행 (n=3). 통계적 유의성 미확보
2. **도메인 편향** — 소프트웨어 설계(RAG)에만 적용. 비소프트웨어 도메인(법률, 의료, 경영)에서의 효과 미검증
3. **변수 미분리** — 이번 실험은 개선된 구조 전체를 적용. 어떤 개선(체크리스트/프레임 검증/대칭 검증)이 가장 기여했는지 분리 불가
4. **Arbiter 모델 일관성** — 주제3에서 Gemini 쿼타 소진으로 Claude가 Arbiter 겸임. 모델 변경이 결과에 영향을 줬을 가능성
5. **재현성** — 온도/시드 미고정. 동일 실험을 재실행하면 다른 결과가 나올 수 있음
6. **설계자 편향** — Executor(Claude)가 실험 설계도 담당. 유리한 방향으로 제안/대응을 구성했을 가능성 배제 불가

### 실무 적용 방향

단독 설계의 기능 구조(멀티에이전트, 관계 그래프 등 **위에 쌓는 기능**)를 유지하면서, 적대적 검증에서 확정된 파이프라인/검색/캐싱 설계를 **아래를 단단하게 만드는 구조**로 적용.

### 기술적 개선 사항 (상세)

### 1. 문서 처리 + 청크 분할 (주제1에서 확정, 8R)

```
(1) 문서 수집
(2) 파싱: 텍스트+레이아웃 동시 추출
    출력: {segments: [{page, section, text, layout_metadata, ocr_confidence}]}
(3) 구간 분석: structure_ratio, table_ratio 자동 계산
    + ocr_confidence 전달 + domain 외부 주입
(4-5) 분할: 속성 기반 경계 결정 → 512토큰 상한
(6) Parent 생성 (~1536) + truncation check
(7) 임베딩+인덱싱 (bge-m3 dense+sparse)
```

- **임베딩**: bge-m3 로컬 (다국어, dense+sparse 하이브리드)
- **검색**: 하이브리드 top-k=10, reranker(로컬) final 3, 메타데이터 필터
- **평가**: 골드셋 50건, 정량 기준 (Recall, ROUGE-L, IoU), 실패 분류 6단계
- **비용**: 로컬 ~$15/월 | API ~$12.50/월

### 2. 검색-생성 아키텍처 (주제2에서 확정, 5R)

**프롬프트**: 시스템(역할+금지+인용규칙) / 컨텍스트(rerank 고점→저점, [Source N] 헤더) / 질문

**컨텍스트 관리**: 4K 예산 분할 (시스템500/질문200/컨텍스트2800/출력500), overflow 시 저점 청크부터 parent→child 축소

**sufficiency gate**: 2단계 임계값

| 조건 | 동작 |
|------|------|
| score < low | 거절 ("관련 정보를 찾을 수 없습니다") |
| low ≤ score < high | 제한 답변 (경고 포함) |
| score ≥ high | 정상 답변 |

**모델 라우팅**: 재작성 질문 + 검색 청크 수 + rerank 최고점 기반. 캐시 히트+high confidence → Haiku 강제.

**인용**:
- 인라인 `[N]` + 하단 매핑 (N ↔ parent_chunk_id)
- span 정보: `{chunk_id, span_start, span_end}`
- span 매칭 3단계 폴백: (1) 정확 매칭 → (2) 편집 거리 > 0.85 → (3) span=null (parent 전체 하이라이트)
- 평가: parent IoU + span IoU 2단계

**스트리밍**: 문장 버퍼 SSE. 전송 조건: (a) 문장 경계 감지 → 문장+인용 메타 전송, (b) EOS → 잔여 flush. 인용 검증: 프롬프트 1차 + 후검증 2차.

**에러 핸들링**: 빈결과 → 안내+재구성 제안, 타임아웃 → Haiku 비스트리밍 재시도, 레이트리밋 → 지수 백오프 3회

**멀티턴**: 토큰 예산 기반 동적 유지(500토큰 이내) + entity_memory. 재작성 후 유사도 < 0.8이면 재검색.

### 3. 캐싱/인덱싱 전략 (주제3에서 확정, 8R)

**3계층 캐시**:

| 계층 | 키 | TTL | 대상 |
|------|-----|-----|------|
| L1 쿼리 (exact) | 정규화+SHA256 | 정적 24h / 동적 1h | 질의→응답 |
| L1 쿼리 (semantic) | 코사인 > 0.95 + 검색설정 해시 | 동일 | 검색 결과만 |
| L2 검색 | reranker 이전/이후 분리 | - | 검색 결과 |
| L3 생성 | 순서 보존 청크 리스트 해시 + 모델 + 프롬프트 버전 | - | LLM 응답 |

- **무효화**: 문서 갱신 시 L1/L2/L3 **동시 무효화**

**인덱싱**:
- **chunk_id** = `doc_id + content_hash` (문서 내 내용 기반)
- **메타데이터 분리 테이블**: chunk_id → [{section_path, section_title, page}] (1:N)
- 구조 변경 → 매핑만 갱신, 임베딩 불변
- 삭제: 매핑 엔트리 제거 → 참조 0개 → 벡터+BM25 삭제 (참조 카운팅)
- 벡터+BM25 일관성: WAL 보장
- 주 1회 full rebuild (단편화 해소, alias 전환)

**모니터링**: 질의당 비용 분해(검색/rerank/생성/캐시), 세그먼트별 적중률, stale 비율, 회귀 벤치마크 50건

**거버넌스**: 민감 정보 캐시 제외, 테넌트별 캐시 분리

## 주제1: RAG 청크 분할 전략과 임베딩 모델 선택

### 초기 제안 (Executor)

- 구조 기반 재귀 분할 512토큰, overlap 64토큰
- bge-m3 로컬 (다국어, dense+sparse)
- 하이브리드 검색 top-k=10, reranker → final 3
- 비용: 로컬 GPU 추론으로 API 비용 0

### Challenger 독립 체크리스트

22항목 생성. 주요 항목:
- CK1: 질문 유형별 성공 기준 정의
- CK2: 입력 문서 구조 분류
- CK8: 표/리스트/코드/수식 직렬화 방식
- CK9: 문서/질의 언어 확인
- CK18: 전체 비용 합산
- CK22: 하이브리드 검색 검토

### 라운드별 진행

| Round | 제기 | 해소 | 핵심 쟁점 |
|-------|------|------|-----------|
| R1 | 9 | C1,C2,C6 | 비용 모순(Cohere+비용0), 저장량 과소(dense만 400MB), 차원 비교 누락 |
| R2 | 6 | C4,C8,C9 | 성공 기준 정량화, TCO 재작성(깨진 문자열), 실패 분류 체계 |
| R3 | 3 | — | 이항 태그의 상호배타 vs 혼합 문서 |
| R4 | 2 | — | 파이프라인에 전처리 단계 누락, 속성-규칙 대응 끊김 |
| R5 | 3 | — | 텍스트 확정 시점과 OCR 후처리 충돌, 레이아웃 정보 소실 |
| R6 | 2 | C15,C16 | 속성 소스 이중성(layout vs 외부), 구간별 스키마 |
| R7 | 2 | C17,C18 | ocr_confidence 이중 정의, 분할 방식과 512토큰 충돌 |
| R8 | 0 | 전부 | Challenger 합의 (rational_consensus) |

### 핵심 전환점

**R3 — "상호배타 분류 폐기했다면서 이항 태그잖아"**

Executor가 상호배타 분류를 폐기하고 속성 태그 기반으로 전환했다고 주장했으나, `[structured|unstructured]×[table_heavy|text_heavy]` 형태로 축 내부는 여전히 이항. Challenger가 이를 지적하면서 파이프라인이 3번 재설계되는 시발점.

**R5 — 레이아웃 정보 보존 문제**

파싱 단계에서 텍스트로 변환하면 레이아웃 정보가 소실되어 구간 분석이 불가능해지는 문제. 해결: 텍스트+레이아웃 동시 추출, 구간 단위 출력 스키마.

### 초기 제안 (Executor)

- 토큰 예산 4K (시스템500/질문200/컨텍스트2800/출력500)

| Round | 제기 | 해소 | 핵심 쟁점 |
|-------|------|------|-----------|
| R1 | 8 | C01,C02,C03,C04,C07 | rerank 정렬 방향, JSON↔스트리밍 충돌, sufficiency gate↔에러 정책 불일치 |
| R2 | 2 | — | SSE 토큰 스트리밍↔문장 인용 시점 충돌, parent↔child 인용 매핑 깨짐 |
| R3 | 4 | — | **프레임 전환 감지**: IoU를 parent로 바꿔 정밀도 저하 은폐 시도 |
| R4 | 3 | — | span 문자열 매칭 취약성, EOS flush 누락 |
| R5 | 0 | 전부 | Challenger 합의 |

### 핵심 전환점

**R3 — 프레임 전환 2회 감지**

1. Executor가 IoU 평가를 child→parent 기준으로 변경하여 정밀도 저하를 비가시화 시도. Arbiter가 `executor_frame_shift_attempted: Detected`로 판정.
2. Executor가 인정하고 2단계 IoU (parent IoU + span IoU)로 기술적 해소.

**R2 — 스트리밍+인용 정합성**

가장 치열했던 쟁점. SSE 토큰 단위 스트리밍과 문장 단위 인용 강제가 동일 시점에 만족 불가. 해결: 문장 버퍼링 방식 SSE.

### 최종 확정 설계

**프롬프트**: 시스템(역할+금지+인용규칙) / 컨텍스트(rerank 고점→저점, [Source N] 헤더) / 질문

**컨텍스트 관리**: 4K 예산 분할, overflow 시 저점 청크부터 parent→child 축소

**sufficiency gate**: 2단계 임계값 (score < low → 거절, low ≤ score < high → 제한 답변, ≥ high → 정상)

**모델 라우팅**: 재작성 질문 + 검색 청크 수 + rerank 최고점 기반. 캐시 히트+high confidence → Haiku 강제.

**인용**:
- 인라인 [N] + 하단 매핑 (N↔parent_chunk_id)
- span 정보: {chunk_id, span_start, span_end}
- span 매칭 3단계 폴백: (1) 정확 매칭 → (2) 편집 거리 > 0.85 → (3) span=null (parent 전체 하이라이트)
- 평가: parent IoU + span IoU 2단계

**스트리밍**: 문장 버퍼 SSE. 전송 조건 2가지: (a) 문장 경계 감지 → 문장+인용 메타 전송, (b) EOS → 잔여 flush (type: flush). 인용 검증은 프롬프트 1차 + 후검증 2차.

**에러 핸들링**: 빈결과 → 안내+재구성 제안, 낮은점수 → 제한답변, 타임아웃 → Haiku 비스트리밍 재시도, 레이트리밋 → 지수 백오프 3회

**멀티턴**: 토큰 예산 기반 동적 유지(500토큰 이내) + entity_memory(핵심 엔터티 독립 유지). 재작성 후 유사도 < 0.8이면 재검색.

## 주제3: RAG 비용 최적화를 위한 캐싱/인덱싱 전략

### 초기 제안 (Executor)

- 3계층 캐시 (L1 쿼리, L2 검색, L3 생성)
- 안정 청크 ID 기반 증분 색인
- 질의당 비용 분해 모니터링

### Challenger 독립 체크리스트

30항목 생성. 주요 항목:
- CK02: 정확 일치 vs 의미 유사 캐시 분리
- CK07: span 인용 캐시 무효화
- CK12: 삭제 전파 메커니즘
- CK13: 안정 청크 ID
- CK22: thundering herd 방지
- CK27: 권한별 캐시 분리

### 라운드별 진행

| Round | 제기 | 해소 | 핵심 쟁점 |
|-------|------|------|-----------|
| R1 | 5 | C1,C3,C4 | L1 TTL이 L2/L3 무효화 덮어쓰기, 안정ID에 position_index 모순, semantic 캐시 키 불충분, L3 순서 무시 |
| R2 | 2 | CK17 | content_hash 동일 청크 충돌, top-k 분기의 순환 전제 |
| R3 | 1 | — | occurrence_index도 선행 삽입/삭제에 취약 |
| R4 | 2 | — | section_hash 범위 문제, offset 연쇄 변경 |
| R5 | 2 | — | 동명 섹션 충돌(WEAK), 구조 변화 미반영 |
| R6 | 2 | — | chunk_id 변경↔메타 갱신 충돌, 동일 ID 복수 경로 |
| R7 | 3 | — | doc_id 포함과 "순수 내용 기반" 모순, 부분 삭제 충돌 |
| R8 | 0 | 전부 | Challenger 합의 |

### 핵심 전환점

**R1~R8 전체 — 청크 ID 안정성 8라운드**

가장 오래 걸린 쟁점. ID 설계가 7번 변경됨:

```
doc_id + section_hash + position_index      (R0 — 초기)
  → position_index가 안정성과 모순 (R1-C2)
doc_id + content_hash(SHA256 앞16자)          (R1 대응)
  → 동일 내용 청크 구분 불가 (R2-C2)
doc_id + content_hash + occurrence_index     (R2 대응)
  → 선행 삽입/삭제에 occurrence 연쇄 변경 (R3-C6)
doc_id + section_title_hash + content_hash + offset  (R3 대응)
  → offset도 선행 편집에 연쇄 변경 (R4-C8)
doc_id + section_title_hash + content_hash   (R4 대응)
  → section_title 변경 시 chunk_id도 변경 (R6-C1)
doc_id + content_hash + 메타데이터 분리 테이블  (R6 대응)
  → doc_id 포함이 "순수 내용 기반" 주장과 모순 (R7-C1)
doc_id + content_hash + 매핑 테이블 + 참조 카운팅 삭제  (R7 대응 — 최종)
```

### 최종 확정 설계

**3계층 캐시**:
- L1 (쿼리): exact (정규화+SHA256, TTL 정적24h/동적1h) + semantic (코사인>0.95, 검색설정 해시 포함, 검색 결과만)
- L2 (검색): reranker 이전/이후 분리 저장
- L3 (생성): 순서 보존 청크 리스트 해시 + 모델 + 프롬프트 버전
- 무효화: 문서 갱신 시 L1/L2/L3 **동시 무효화**

**인덱싱**:
- chunk_id = doc_id + content_hash (문서 내 내용 기반)
- 메타데이터 분리 테이블: chunk_id → [{section_path, section_title, page}] (1:N)
- 구조 변경 → 매핑만 갱신, 임베딩 불변
- 삭제: 매핑 엔트리 제거 → 0개 되면 벡터+BM25 삭제 (참조 카운팅)
- 벡터+BM25 일관성: WAL 보장
- 주 1회 full rebuild (단편화 해소 목적, alias 전환)

**질의 적응**: 재작성 질문 규칙 기반으로 top-k/rerank depth 분기 (순환 전제 없음)

**모니터링**: 질의당 비용 분해(검색/rerank/생성/캐시), 세그먼트별 적중률, stale 비율, 회귀 벤치마크 50건

**거버넌스**: 민감 정보 캐시 제외, 테넌트별 캐시 분리

## 단독설계 RAG와의 비교

비교 대상: `interview-wiki-agents` (LangGraph 기반 멀티에이전트 면접위키 Q&A)

### 핵심 비교

| 영역 | 단독설계 | 적대적 검증 |
|------|---------|------------|
| 분할 | Markdown 헤더 + Recursive, 모든 문서 동일 | 구간 단위 속성 기반 적응 분할 |
| 임베딩 | text-embedding-3-small (API, 1536차원) | bge-m3 (로컬, 1024차원, dense+sparse) |
| 검색 | 코사인 단일, top-k=5 고정 | 하이브리드 + reranker + 적응 top-k |
| 인용 | 없음 | span 3단계 폴백 + 2단계 IoU |
| 환각 방지 | 프롬프트 수준 | sufficiency gate + 후검증 |
| 캐싱 | 없음 | 3계층 + 전계층 무효화 |
| 증분 색인 | 없음 (전체 삭제→재삽입) | content_hash 기반 + 메타 분리 |
| 비용 | API 종속 (통제 불가) | 로컬 우선 ~$15/월 |
| 멀티에이전트 | Router → 4개 전문 에이전트 | 없음 (단일 파이프라인) |
| 관계 그래프 | topic_relations 테이블 | 없음 |
| 기능 다양성 | search/quiz/explain/compare | Q&A 단일 |

상세 실무/학술 결론은 문서 상단 참조.
