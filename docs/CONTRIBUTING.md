# 실험 및 문서 작성 가이드

새 실험을 수행하거나 개선 사항을 기록할 때의 규칙.

## 실험 기록 작성

### 폴더 구조

```
docs/experiments/YYYY-MM-DD-<주제>/
├── README.md              # 실험 기록 본문 (결론 → 설계 → 리서치)
└── improvements/
    ├── 01-<개선사항>.md
    ├── 02-<개선사항>.md
    └── ...
```

### 문서 구조 (두괄식)

```markdown
# <주제> 적대적 검증 실험 (YYYY-MM-DD)

> 한 줄 요약

---

## 결론

### 실무 결론
- 개선 전 vs 개선 후 비교 테이블
- 핵심 한 줄 요약

### 검증된 것
- 번호 목록으로 나열

### 검증되지 않은 것 (한계)
- 번호 목록으로 나열

### 실무 적용 방향
- 1~2문장

---

## 최종 확정 설계

### 기술적 개선 사항 (상세)
- 개선 사항 테이블 + docs/improvements/ 링크

### 확정 설계 상세
- 각 소주제별 확정 내용

---

## 리서치 상세

## 실험 환경
- 모델, 호출 방식, 파라미터

## 주제별 라운드 기록
- 초기 제안 → 체크리스트 → 라운드별 진행 → 핵심 전환점 → 최종 설계

## 시스템 검증 결과
- 구조별 작동 여부 테이블
- 이전 실험 대비 개선 테이블
```

### 원칙

1. **두괄식**: 결론 → 확정 설계 → 리서치 상세 순서
2. **비교 테이블 필수**: 개선 전/후가 한눈에 보여야 함
3. **한계 명시**: 검증된 것만큼 검증되지 않은 것도 중요
4. **링크 연결**: 개선 사항은 improvements/ 개별 문서로 분리

---

## 기술적 개선 사항 작성

### 파일 위치

```
docs/experiments/YYYY-MM-DD-<주제>/improvements/<번호>-<이름>.md
```

### 문서 구조

```markdown
# <개선 사항 이름>

> 어떤 실험에서, 몇 라운드에 걸쳐 확정

## 문제
- 무엇이 문제였는지 (1~2문장)

## 단독 설계에서는
- 이 문제가 왜 발견되지 않았는지

## 적대적 검증에서 일어난 일
- 라운드별로 어떤 논의가 있었는지
- Challenger의 핵심 지적
- 프레임 전환이 감지되었다면 명시

## 최종 설계
- 확정된 설계 내용 (코드/다이어그램 포함 가능)

## 개선된 점
- 개선 전 vs 개선 후 비교 테이블 (필수)

## 왜 이 문제가 중요한가 / 왜 단독 설계에서 못 찾았나
- 1~2문장 설명
```

### 원칙

1. **문제 → 논의 → 해소 → 결과** 순서 고정
2. **비교 테이블 필수**: 개선 전/후
3. **프레임 전환 감지 시 명시**: 어떤 시도가 차단되었는지
4. **단독 설계와의 차이**: 왜 이 시스템이 필요한지를 보여주는 핵심

## README 업데이트

새 실험/개선 사항 추가 시 README.md에도 링크 추가:

```markdown
### 실험 기록

- **[새 실험](docs/experiments/YYYY-MM-DD-<주제>/)** — 요약
  - [개선1](docs/experiments/YYYY-MM-DD-<주제>/improvements/<번호>-<이름>.md) — 한줄 설명
```

## 커밋 메시지 규칙

```
feat: 새 실험/개선 사항 추가
refactor: 기존 문서 구조 변경
fix: 오류 수정
```

본문에 변경 사항 불릿 목록 포함.
