# 인용 정합성

> 주제2 (검색-생성) — 5라운드에 걸쳐 3단계 폴백 + 2단계 IoU 확정

## 문제

LLM이 "문서에 따르면..."이라고 답변하면서 **실제로는 문서에 없는 내용을 인용**하거나, 인용 위치가 **실제 근거와 다른 곳**을 가리키는 문제.

## 단독 설계에서는

인용 기능 자체가 없음. 사용자가 "이 답변의 근거가 뭐야?"라고 물어봐도 확인할 방법이 없음.

## 적대적 검증에서 일어난 일

**R1** — Challenger: "인용 [1]이 가리키는 청크와 실제 답변 근거가 같다는 보장이 없다"

**R2** — Challenger: "SSE 토큰 스트리밍에서 문장 단위 인용을 강제하면, 문장이 완성되기 전에 토큰이 전송된다. 인용 시점과 전송 시점이 충돌한다"

**R3** — Challenger: "Executor가 IoU 평가를 child→parent 기준으로 바꿔서 정밀도 저하를 은폐하려 했다" → **프레임 전환 감지** → Executor 인정 → 2단계 IoU로 재설계

**R4** — Challenger: "span 문자열 정확 매칭은 OCR 오류/정규화 차이에서 실패한다"

## 최종 설계

**span 매칭 3단계 폴백**:
1. 정확 매칭 → 성공 시 정확한 하이라이트
2. 편집 거리 > 0.85 → 유사 구간 하이라이트
3. span=null → parent 청크 전체 하이라이트 (최후 수단)

**2단계 IoU 평가**:
- parent IoU: 올바른 문서/페이지를 가리키는가?
- span IoU: 올바른 문장/구간을 가리키는가?

## 개선된 점

| 항목 | 개선 전 | 개선 후 |
|------|--------|--------|
| 인용 존재 | 없음 | 인라인 `[N]` + 하단 매핑 |
| 인용 정확도 검증 | 없음 | 프롬프트 1차 + 후검증 2차 |
| 인용 실패 시 | 잘못된 인용 그대로 노출 | 3단계 폴백으로 최선의 하이라이트 보장 |
| 평가 지표 | 없음 | parent IoU + span IoU |

## 프레임 전환이 감지된 지점

Executor가 R3에서 IoU 평가 기준을 child→parent로 바꿔 정밀도 저하를 비가시화하려 했음. Arbiter가 `executor_frame_shift_attempted: Detected`로 판정. 이것이 없었다면 **정밀도가 낮은 인용 시스템이 "검증 통과"로 포장**되었을 것.
