# 청크 ID 안정성

> 주제3 (캐싱/인덱싱) — 8라운드에 걸쳐 7번 재설계

## 문제

문서가 수정되면 청크의 내용은 같은데 **ID가 바뀌어** 벡터 DB와 BM25 인덱스 전체를 재구축해야 하는 비용 폭증 문제.

## 단독 설계에서는

`doc_id + position_index`로 끝. 문서 중간에 한 줄 추가하면 **그 아래 모든 청크의 ID가 변경**됨. 이 문제를 인지조차 못 함.

## 적대적 검증에서 일어난 일

Challenger가 **7번 연속으로 결함을 발견**하면서 ID 설계가 점진적으로 개선됨:

```
R0: doc_id + section_hash + position_index
    → "position_index가 안정적이라는 근거가 없다" (R1-C2)

R1: doc_id + content_hash(SHA256 앞16자)
    → "동일 내용 청크가 여러 개면 구분 불가" (R2-C2)

R2: doc_id + content_hash + occurrence_index
    → "앞에 삽입하면 occurrence도 밀린다" (R3-C6)

R3: doc_id + section_title_hash + content_hash + offset
    → "offset도 편집하면 밀린다" (R4-C8)

R4: doc_id + section_title_hash + content_hash
    → "section_title이 바뀌면 ID도 바뀐다" (R6-C1)

R6: doc_id + content_hash + 메타데이터 분리 테이블
    → "doc_id를 포함하면서 순수 내용 기반이라는 주장은 모순" (R7-C1)

R7 (최종): doc_id + content_hash + 매핑 테이블 + 참조 카운팅 삭제
```

## 개선된 점

| 항목 | 개선 전 | 개선 후 |
|------|--------|--------|
| 문서 수정 시 | 전체 청크 ID 변경 → 전체 재인덱싱 | **내용 변경된 청크만** ID 변경 |
| 구조 변경 시 (제목 수정 등) | ID 변경 + 임베딩 재계산 | **매핑만 갱신**, 임베딩 불변 |
| 삭제 시 | 고아 벡터 잔존 | **참조 카운팅**으로 0이 되면 자동 삭제 |
| 동일 내용 청크 | 구분 불가 또는 불안정 | content_hash로 자연 병합 |

## 이 문제를 왜 단독 설계에서 못 찾았나

ID 설계는 **시스템이 정상 동작할 때는 보이지 않는 문제**. 문서가 수정되는 순간에야 비로소 드러나는데, 설계 단계에서는 "수정 시 어떻게 되지?"를 질문할 동기가 없음. Challenger가 이 질문을 **강제로 던졌기 때문에** 발견됨.
